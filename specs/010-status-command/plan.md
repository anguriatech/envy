# Implementation Plan: Vault Sync Status Command

**Branch**: `010-status-command` | **Date**: 2026-03-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/010-status-command/spec.md`

## Summary

Implement `envy status`, a zero-latency, passphrase-free command that shows the sync state between the local vault and the `envy.enc` artifact. Requires a schema V1→V2 migration (new `sync_markers` table), wiring of `seal_env` to write a sync timestamp after every successful encrypt, a DB aggregation query, and a CLI command rendering a table or JSON report.

## Technical Context

**Language/Version**: Rust stable (edition 2024, MSRV 1.85)
**Primary Dependencies**: `rusqlite` (bundled-sqlcipher), `clap` (derive), `serde_json`, `comfy-table = "7"` *(new)*
**Storage**: SQLite via `bundled-sqlcipher-vendored-openssl` (existing vault) + `sync_markers` table (new, V2 schema)
**Testing**: `cargo test` (unit + integration); existing `tempfile` dev-dependency
**Target Platform**: Linux / macOS / Windows (same as existing CI matrix)
**Project Type**: CLI tool
**Performance Goals**: `envy status` completes under 500 ms for 100 envs / 1 000 secrets (SC-001); no crypto required, so only DB aggregation latency matters
**Constraints**: MUST NOT decrypt any envelope; MUST NOT prompt for passphrase; read-only w.r.t. secrets
**Scale/Scope**: Single-user local CLI; vault typically has 1–20 environments, 10–200 secrets total

## Milestone 1: Schema + Database Layer

**Goal**: The `sync_markers` table exists in all vaults (new and migrated); the DB layer can upsert a marker and query aggregate status.

### 1.1 Schema V2 Migration (`src/db/schema.rs`)

Add `SCHEMA_V2` constant:
```sql
CREATE TABLE IF NOT EXISTS sync_markers (
    environment_id  TEXT    NOT NULL PRIMARY KEY
                            REFERENCES environments(id) ON DELETE CASCADE,
    sealed_at       INTEGER NOT NULL
);
```

Extend `run_migrations`:
```rust
// existing:
if version == 0 { apply SCHEMA_V1; set user_version = 1; }
// new:
if version == 1 { apply SCHEMA_V2; set user_version = 2; }
```

Migration is additive. Existing V1 vaults get an empty `sync_markers` table; their environments display "Never Sealed" until re-encrypted.

Schema version comment block at the top of the file must be updated:
```
//!   1 → 2: Add sync_markers table (sealed_at per environment).
```

### 1.2 Sync Marker Upsert (`src/db/sync_markers.rs` — new file)

```rust
impl Vault {
    pub fn upsert_sync_marker(&self, env_id: &EnvId, sealed_at: i64) -> Result<(), DbError>
```

SQL: `INSERT OR REPLACE INTO sync_markers (environment_id, sealed_at) VALUES (?1, ?2)`

### 1.3 Environment Status Aggregation (`src/db/sync_markers.rs`)

```rust
pub struct EnvironmentStatus {
    pub name: String,
    pub secret_count: i64,
    pub last_modified_at: Option<i64>,  // MAX(secrets.updated_at); None if 0 secrets
    pub sealed_at: Option<i64>,         // NULL if never sealed
}

impl Vault {
    pub fn environment_status(&self, project_id: &ProjectId) -> Result<Vec<EnvironmentStatus>, DbError>
```

SQL:
```sql
SELECT
    e.name,
    COUNT(s.id)          AS secret_count,
    MAX(s.updated_at)    AS last_modified_at,
    sm.sealed_at
FROM environments e
LEFT JOIN secrets s  ON s.environment_id = e.id
LEFT JOIN sync_markers sm ON sm.environment_id = e.id
WHERE e.project_id = ?1
GROUP BY e.id, e.name, sm.sealed_at
ORDER BY e.name ASC
```

`MAX(s.updated_at)` returns NULL when there are no secrets (SQLite aggregate NULL rule); the Rust mapping uses `Option<i64>` for that column.

### 1.4 DB Module Re-exports (`src/db/mod.rs`)

Add:
```rust
mod sync_markers;
pub use sync_markers::EnvironmentStatus;
```

### 1.5 DB Unit Tests

In `src/db/sync_markers.rs` (under `#[cfg(test)]`):
- `schema_v2_migration_adds_sync_markers_table`: open vault, assert `table_exists("sync_markers")` returns `true`
- `upsert_sync_marker_creates_row`: create env, upsert marker, query `SELECT sealed_at FROM sync_markers`
- `upsert_sync_marker_updates_existing_row`: upsert twice, assert only one row with the second timestamp
- `environment_status_returns_never_sealed_when_no_marker`: create env + secret, no sync marker → `sealed_at = None`

- `environment_status_zero_secrets`: env with no secrets → `secret_count == 0`, `last_modified_at == None`
- `sync_marker_deleted_on_environment_cascade`: delete env, assert sync_markers row gone

## Milestone 2: Core Layer — SyncStatus Derivation + seal_env Wiring

**Goal**: `seal_env` writes a sync marker after every successful seal; core exports a `get_status_report` function for the CLI.

### 2.1 SyncStatus Enum and Derivation (`src/core/status.rs` — new file)

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum SyncStatus {
    InSync,
    Modified,
    NeverSealed,
}

pub struct StatusRow {
    pub name: String,
    pub secret_count: i64,
    pub last_modified_at: Option<i64>,
    pub sealed_at: Option<i64>,
    pub sync_status: SyncStatus,
}
```

`derive_sync_status(secret_count, last_modified_at, sealed_at) -> SyncStatus` logic:
```
sealed_at is None            → NeverSealed
last_modified_at is None     → InSync (no secrets, marker exists — treated as in-sync)
last_modified_at > sealed_at → Modified
else                         → InSync
```

`get_status_report(vault, project_id) -> Result<Vec<StatusRow>, CoreError>`:
- Calls `vault.environment_status(project_id)` (DB layer)
- Maps each `EnvironmentStatus` → `StatusRow` by calling `derive_sync_status`
- Returns sorted Vec (already sorted by DB query)

### 2.2 Seal Env Wiring (`src/core/sync.rs`)

After `Ok(seal_envelope(passphrase, &payload)?)` in `seal_env`, add the sync marker upsert:

```rust
// Record the seal timestamp for `envy status` sync tracking (FR-008).
let env_id = vault
    .get_environment_by_name(project_id, env_name)
    .map_err(|e| SyncError::VaultError(e.to_string()))?
    .id;
let now = std::time::SystemTime::now()
    .duration_since(std::time::UNIX_EPOCH)
    .unwrap_or_default()  // SAFETY: system clock is always after UNIX_EPOCH
    .as_secs() as i64;
vault
    .upsert_sync_marker(&env_id, now)
    .map_err(|e| SyncError::VaultError(e.to_string()))?;
```

The `unwrap_or_default()` on `duration_since` is safe because `UNIX_EPOCH` is the minimum possible `SystemTime` on all supported platforms; this is documented inline.

### 2.3 Core Module Re-exports (`src/core/mod.rs`)

Add:
```rust
pub mod status;
pub use status::{SyncStatus, StatusRow, get_status_report};
```

### 2.4 Core Unit Tests

In `src/core/status.rs` (under `#[cfg(test)]`):
- `derive_sync_status_never_sealed`: `sealed_at = None` → `NeverSealed`
- `derive_sync_status_in_sync`: `sealed_at >= last_modified_at` → `InSync`
- `derive_sync_status_modified`: `last_modified_at > sealed_at` → `Modified`
- `derive_sync_status_no_secrets_with_marker`: `last_modified_at = None, sealed_at = Some(t)` → `InSync`
- `seal_env_writes_sync_marker`: call `seal_env`, then `vault.environment_status`, assert `sealed_at` is Some and recent

### 3.2 Clap Command (`src/cli/mod.rs`)

```rust
/// Show sync status between the local vault and the envy.enc artifact.
///
/// Displays environment names, secret counts, last-modified times, and
/// whether each environment is in sync with the sealed artifact.
/// Never prompts for a passphrase or decrypts any secret payloads.

```

Add dispatch in `run()`:
```rust
Commands::Status => {
    let artifact = artifact_path(&manifest_path);
    match commands::cmd_status(&vault, &project_id, &artifact, cli.format) {
        Ok(()) => 0,
        Err(e) => {
            eprintln!("{}", format_cli_error(&e));
            cli_exit_code(&e)
        }
    }
}
```

### 3.3 Command Handler (`src/cli/commands.rs`)

Signature:
```rust
pub(super) fn cmd_status(
    vault: &Vault,
    project_id: &ProjectId,
    artifact_path: &Path,
    format: OutputFormat,
) -> Result<(), CliError>
```

#### Table rendering path

1. Call `crate::core::get_status_report(vault, project_id)` → `Vec<StatusRow>`
2. If empty → print "No environments found..." and proceed to artifact section.
3. Build `comfy_table::Table` with header `["Environment", "Secrets", "Last Modified", "Status"]`.
4. For each `StatusRow`: format `last_modified_at` via `humanize_timestamp`, colorize status cell.
5. `println!("{table}")`.
6. Print artifact section (see §3.4).

#### JSON rendering path

1. Call `crate::core::get_status_report(vault, project_id)`.
2. Build `StatusJson` struct with `environments` array + `artifact` object.
3. `serde_json::to_writer(&mut stdout(), &payload)?; writeln!(stdout())?;`

#### `humanize_timestamp(epoch: i64) -> String`

Private function in `commands.rs`:
```
diff = now_unix - epoch
< 60   → "X seconds ago"
< 3600  → "X minutes ago"
< 86400 → "X hours ago"
< 604800 → "X days ago"
else   → format YYYY-MM-DD from epoch
```
For `epoch = 0` or `diff < 0` (future timestamp or unset): return `"unknown"`.

### 3.4 Artifact Section Rendering

```rust
fn render_artifact_section(artifact_path: &Path) -> String
```

Reads `envy.enc`:
- If missing: `"Artifact: {path}  — not found"`
- If malformed: `"Artifact: {path}  — unreadable (malformed JSON)"`
- If valid: `"Artifact: {path}  (last written: {humanize})\n  Sealed environments: {names}"`
- Extra: if any name in artifact is not in vault status rows → warn `"  ⚠  {name} is in the artifact but not in the local vault"`

Reading artifact names: deserialize via `crate::core::read_artifact(path)` (already exists); extract `.environments.keys()`. No passphrase, no decryption.

## Constitution Check (Post-Design)

| Principle | Verification |
|-----------|-------------|
| I. Security | `cmd_status` only calls `get_status_report` (no crypto); artifact parsing reads only JSON keys (no `unseal_envelope`). Sync marker stores only a timestamp — no secret material ever reaches `sync_markers`. |
| II. Determinism | SQL query is deterministic (`ORDER BY e.name ASC`); relative-time formatting is a pure function of current time (documented non-determinism). JSON output field order is stable via struct field order in Serde. |
| III. Rust Best Practices | All new functions: `Result<T, E>`. One permitted `unwrap_or_default()` in `seal_env` with inline justification. Unit tests required at DB layer, Core layer, and CLI layer. |
| IV. Modularity | DB layer: raw numbers only. Core layer: SyncStatus derivation, no rendering. CLI layer: rendering only, no DB access. `cmd_status` is the sole composition point. |
| V. Language | All identifiers, comments, user-facing strings in English. |
